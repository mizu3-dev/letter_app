# 「手紙ポスト」アプリ Flutter×Supabase×Cloudflare 統合仕様書（最終完全版）

本ドキュメントは、「手紙ポスト」アプリにおけるローカルモック環境からバックエンドプラットフォーム（SupabaseおよびCloudflare R2）への移行設計、ワイヤーフレームに基づく情緒的なUI/UX演出、新設される「フレンドタブ」「交換日記リレー機能」に加え、**オリジナルフォントの最適化・作成フローの高度化、文字データ通信制限、Supabase/Cloudflareのセキュリティ・コスト最適化設計、手紙作成画面のUI詳細、エッジケース対策、グループ送信機能拡張（1人限定受信）、画像アセットの動的差し替え、テーマカラーの拡張性、近代SNSライクな操作性、レーダー・交換日記の拡張仕様、および本番用マルチ認証システム（各種SNS・メアド認証、デバッグ時限定の匿名認証）**のすべてを内包した、実装・開発用の最終統合仕様書です。

---

## 1. アプリケーション概要と基本仕様

「手紙ポスト」は、即時性を重視する現代のチャットアプリとは対照的に、あえて手紙が届くまでに時間を要する（配送時間を要する）「スローライフ」なコミュニケーションを提供するSNSです。
デジタルでありながら、本物の紙やペン、ノートを使ったような温かみ、おしゃれな質感、および届くまでのワクワク感を演出する情緒的なUI/UXを最大の特徴とします。

### 1.1 主要開発言語・技術スタックとテーマ拡張性
* **クライアントサイド**: **Flutter (Dart)** を主軸として開発します。マルチプラットフォーム（iOS / Android）において、本仕様書で定義するリッチな全画面アニメーションや、文字ごとの動的スタイル描画、手書きキャンバスのストローク制御をスムーズに実現します。
* **拡張性のあるカラーシステム設計**: テーマカラーについては、現在は「ライトモード」と「ダークモード」のみですが、将来的にサクラテーマ、レトロテーマ、パステルテーマなどの新しいカラーテーマを柔軟に追加・拡張できるよう、`ThemeData` を動的にインジェクション可能な設計（`ThemeExtensions` の活用）とします。
* **バックエンド・インフラ（無料枠・コスト最適化構成）**: 高い無料枠と拡張性を活かすため、**Supabase** と **Cloudflare R2** を全面的に採用します。
  * **Supabase Authによるマルチ認証**: 本番環境ではセキュリティと信頼性を担保するため、**メールアドレス/パスワード認証、Google認証、Microsoft認証、GitHub認証、Apple認証**の5つの堅牢な認証方法を標準提供します。開発およびデバッグ効率化のため、デバッグビルド時に限り「匿名認証（Anonymous Auth）」による仮サインインを限定許容しますが、本番ビルドでは匿名認証への動線を完全に遮断・無効化する構成とします。
  * **Supabase Database & Realtime**: PostgreSQLベースの堅牢なデータ管理と、Row Level Security (RLS) による強力なアクセス制御、および配送状況のリアルタイム同期を行います。
  * **Cloudflare R2**: 容量制約を打破し通信コストを抑えるため、大容量データ（添付写真、オリジナルフォントファイルなど）の保管には無料枠（10GB/月）が非常に寛容なCloudflare R2（S3互換ストレージ）を採用します。
  * **プッシュ通知基盤**: プッシュ通知の配信には、OSネイティブの通知ゲートウェイと連携する配信基盤を利用します。Supabase Edge Functions または Cloudflare Workers から通知APIを呼び出すことで、効率的かつ確実なバックグラウンド通知を実現します。

### 1.2 デザインコンセプトと近代SNSライクな操作性
* **近代SNSライクな使い勝手の融合**: 全体的にアナログな温かみを残しつつも、画面間のスワイプ遷移、インラインでのローディングシマー（Shimmer）効果、長押しコンテキストメニュー、ドラッグ＆ドロップによる直感的な並び替えなど、**近代的なSNSと同等のサクサクとした、ストレスのない優れたユーザー体験（UX）**を両立させます。
* **画像アセットの後日差し替え（分離）仕様**: 便箋、封筒、シール、各種バッジ、配達員イラストなどのデザインパーツは、後日デザインチームが作成するオリジナル画像（PNG/SVG）へ容易に差し替えられるよう、ハードコーディングを禁止します。すべて一元管理されたアセット管理クラス（`AssetManager`）を介して、ID（String）に基づき動的にパスを呼び出す構造とします。

### 1.3 主要機能と画面構成・情緒的演出仕様
下部ナビゲーションには以下の5つのタブを配置します。中央の「手紙作成」ボタンは丸く大きく際立たせたモダンなUIとし、タップ時の挙動に特殊な画面遷移を導入します。
* **配置**: `ポスト` / `レーダー` / `手紙作成`（中央・丸く大きく配置） / `フレンド` / `マイページ（設定含む）`

#### ① ポスト（受信箱・送信履歴・配送ステータス表示・交換日記閲覧）
* **情緒的な配送ステータス（「てがみをまつ」演出）**: 
    * 配送中の手紙・交換日記は、無機質な秒単位のカウントダウンではなく、「約h時間後」のように大まかに表現します。
    * 画面上には、可愛い配達員や車のイラストとともに、現在の状況が「向かい中…もう少し！」「しわけ中 まだまだ」といった情緒的な言葉で可視化され、手紙を待つ時間を楽しく演出します。
* **「うけとる」・開封・破棄演出**:
    * 手紙や交換日記が届くと「手紙がとどきました！」という通知とともに、画面上に味のある「ポスト」が表示されます。
    * 届いたアイテムは「あける」または「すてる」を選択できます。「あける」を選択すると、実際に封筒から手紙を取り出したり、日記帳の表紙を開いたりするような、おしゃれな開封アニメーション演出が入ります。
    * 「すてる」を選択すると、コミカルかつ情緒的なエフェクトとともにアイテムがごみ箱に吸い込まれていくアニメーション演出が入り、ユーザーの画面から破棄されます。
* **受信アイテムの描画と過去の筆跡維持**:
    * 送信者が指定した便箋/日記テーマ、文字サイズ、文字色、フォント（既定、または手書きオリジナル）で美しく描画されます。
    * 送信者が手紙を送信した「その時点の筆跡」を正確に維持するため、データには送信時のフォントの**バージョン情報（リビジョン番号）**を記録し、送信後にフォントが更新されても過去の筆跡が勝手に書き換わらないように保証します。
* **交換日記のリレーハブ機能**:
    * 自分の順番が回ってきた交換日記がポストに届いた際、これまでの全員の記録を「本（見開きノート）のような質感」でパラパラとページをめくって閲覧できる特別なブックビューアUIを提供します。
    * ポスト内の交換日記詳細画面に「**次の人へ書いて送信**」ボタンを配置し、タップするとシームレスに手紙作成と同じフロー（全画面エディタ）へと遷移します。
    * 「秘密の話」や「グッドニュース」「バッドニュース」「質問相談コーナー」「恋バナ」など、一周回るごとに話題のテーマ（お題）がシステムから与えられます。
* **交換日記の執筆滞留（ストップ）対策（UX維持仕様）**:
    * リレー形式の交換日記が特定のユーザーで止まるのを防ぐため、自分の番になってから**48時間**以上執筆がない場合、システムが該当ユーザーへ自動で催促通知を送信します。
    * **72時間**が経過した場合、日記のステータスを一時的に `overdue`（期限切れ）に変更し、グループの管理者（オーナー）または他のメンバーの画面（フレンドタブ内のグループ管理画面）に「現在の執筆者をスキップして次の人へ日記を強制転送する」ボタンを表示します。これにより、グループ全体の機能停止を防止します。

#### ② レーダー（置き手紙レーダー）
* **探索型UI**: フレンドまたは所属グループのメンバーが、特定のGPS位置に設置した「置き手紙（Drop Letter）」を探すための機能です。ターゲットへ近づくほど徐々に情報が正確になっていく「探索型UI」を採用し、宝探しのようなワクワク感を演出します。
* **通知仕様**: フレンドまたは所属グループの誰かが置き手紙を設置した瞬間、対象メンバーへ「置き手紙が設置されたよ！」というプッシュ通知が届きます。レーダー画面を開くと、プロットされた大まかな場所の情報が地図上に可視化されます。
* **コア仕様**: ユーザー（自分）を中心として、手紙が存在する相対的な「方向」と「距離」をレーダー風の画面上にリアルタイムにプロットします。距離が離れているときは大まかに「方向」のみが波紋のように表示され、距離が近づくにつれて明確な「点（ピン）」としてプロットされます。実際に端末を持って移動し、設置場所から**3m以内**の至近距離に達したとき、初めて手紙の「発見・開封」が可能になります。
* **接近通知とリアルタイムフィードバック**: スマートフォンが置き手紙の一定距離（**50m以内**）に進入すると、バックグラウンド状態であっても通知・振動（バイブレーション）・音でユーザーに接近を知らせます。50m以内に入った時点から、画面上には「あと○○m！」という秒単位の正確な距離の減算カウントと、近代SNSライクな流暢なリアルタイムフィードバックが実行されます。

#### ③ 手紙作成 / 交換日記執筆（イマーシブ・全画面エディタ）の詳細UI仕様
下部ナビゲーション中央の丸く大きな「手紙作成」ボタン、またはポストの「次の人へ書いて送信」ボタンを押した瞬間から、手紙がポストへ投函されるまでの詳細なUI/UX遷移を以下に定義します。

1. **没入型（イマーシブ）画面遷移**:
    * ボタンがタップされた瞬間、**下部ナビゲーションバーが画面下方へスムーズにスライドアウトして完全消失**します。
    * 同時に、画面全体が執筆専用のエディタ背景へとシームレスに切り替わり、ユーザーが手紙の執筆に100%没頭できるフルスクリーン空間へと誘います。

2. **「かく（紙選び）」のUI詳細**:
    * 最初に便箋（かみえらび）画面が起動します。画面中央にレターセットの束がリアルな影を伴う質感で配置されます。
    * 左右へのスワイプジェスチャーにより、本物の紙を「ベラベラ、パラパラ」とめくって選ぶような、心地よいスクロールエフェクトと微細な紙の摩擦音（SE）が走ります。
    * 各便箋には、温かみのあるクラフト紙、淡いサクラ色、夜空を模した濃紺など情緒的なテーマが割り当てられており、お気に入りを選んで下部の「この便箋にする」をタップして決定します。

3. **「かき中（エディタ・部分スタイル適用）」のUI詳細**:
    * **画面構成**: 決定した便箋テーマ（罫線や地紋など）が画面全面の背景に敷かれ、その上にスクロール可能なテキスト入力エリアが広がります。
    * **データベース制限に伴うソフトリミット**: データ容量の逼迫やメタデータ肥大化によるバグを防ぐため、アプリのエディタ側で「最大30,000文字まで」、および「スタイル変更（リッチテキストの区切れ）は最大200箇所まで」という実用上十分なソフト制限（バリデーション）を実装します。上限に達しそうな場合は「少し手紙が長すぎるようです。新しいページに分けましょう」等の案内を表示します。
    * **上部固定ツールバー**: 以下の3つのトグルボタンが配置されます。
        * *ペン（フォント選択）*: 「既定のゴシック」「既定の明朝」「手書き風」「自分のオリジナルフォント」を切り替えます。
        * *色（カラーパレット）*: 墨、セピア、藍色、茜色といった、便箋に馴染む落ち着いた色のインクを選択できます。
        * *文字サイズ*: 小・中・大・特大を切り替えるステップトグル。
    * **部分スタイル（個別フォント）適用機能**: テキスト入力中、任意の文字列を長押ししてドラッグ選択すると、近代SNS同様のフローティングポップアップメニューが出現します。ここから特定の文字や単語、1文章のブロック単位に対してのみ、フォント・色・サイズを個別に変更できます。「手紙の基本は明朝体で書き、感情を高ぶらせたい一言だけを自分のオリジナルフォント（手書き）と大きな赤い文字に変える」といったリッチな表現が可能です。
    * **軽量な文字データ通信（画像化の禁止）**: 通信コストおよびデータベース容量を最小限に抑えるため、手紙データはすべて純粋な「文字（テキスト）データ」および「スタイル・位置のメタデータJSON」として送信します。画面の画像化（ラスタライズデータの送信）は行いません。
    * **画像添付スロット**: 便箋の最下部（または改ページ直前）に固定の「＋」スロットが1箇所配置されます。タップするとカメラまたはギャラリーが起動します。写真はポラロイドカメラ風の白枠フレームに自動加工され、便箋の上に**1枚のみ**美しくレイアウトされます。
    * **改ページUI**: 「＋ページを追加」ボタンをタップすると、現在の便箋の下に新しいページがスライドインで追加されます。ページごとに異なる便箋デザインを「紙選び」から再選択することが可能です。

4. **「送信前の不足文字その場作成ステップ」のUI詳細**:
    * ユーザーが執筆を終えて「かきおわった！」ボタンをタップすると、システムが本文テキストを走査し、ユーザーが`まだオリジナルフォントに登録していない未作成の文字`が含まれているかどうかを瞬時に検出します。
    * 未登録の文字が検出された場合、自動的に全画面のステップ画面が起動し、「手紙を相手に届けるために、あと○文字だけあなたの手を貸してください」というメッセージが表示されます。
    * **文字作成のUI構成**:
        * *上部（お手本表示）*: システムフォントで「次に書く対象の文字（例：『鬱』『☆』）」が1文字、画面中央上部に大きくプレビューされます。
        * *中央（手書きキャンバス）*: 画面の横幅いっぱいに、**「大きな正方形（1:1）の手書きキャンバスエリア」**が配置されます。キャンバス内には、文字のバランスを取りやすくするための薄い点線の十字ガイド（十文字の補助線）が敷かれています。ユーザーはこの四角形の中に直接文字を肉筆で書き込みます。
        * *下部（コントロール）*: 「書き直す（クリア）」ボタンと「できた！」ボタンが配置されます。
        * **ストローク数バリデーションと平滑化（スムージング）**: キャンバスの描画データ（パス）が「0ストローク（白紙）」の場合は「できた！」ボタンを非活性化します。また、あまりにも複雑な塗りつぶしの場合はデータ肥大化防止のため「もう少しシンプルに書いてください」と制限をかけます。指で書いた線のガタつきを綺麗に見せるため、Flutter側でサンプリング点の補間（ベジエ曲線化・平滑化フィルター）を施してからSupabaseへベクター保存し、おしゃれな質感とデータ軽量化を両立します。
    * すべての不足文字の書き込みが完了すると、自動的にフォントのバージョンがインクリメントされ、Supabase Databaseへベクターデータ（SVG形式）がアップロードされた後、次の封緘（ふうかん）フェーズへ自動で遷移します。

5. **「かきおわり・しまう演出（封緘）」のUI詳細**:
    * カルーセル画面から手紙を包む「ふうとう（封筒）」と、留める「シール」を選択します。
    * 「封をする」をタップすると、それまで広がっていた便箋がアニメーションによって美しく三つ折りに折りたたまれていき、選んだ封筒の口へ「シュッ」と吸い込まれるように収まります。その後、封筒の蓋が閉じ、選んだシールが「ポン」と小気味よいエフェクトとともに貼り付けられ、「てがみ かんせい！」状態の封筒が画面中央に直立するリアルなフルアニメーションが実行されます。

6. **「おくる（配送選択・グループ1人限定受信オプション）」のUI詳細**:
    * 完成した封筒の下部に、配送手段を選択する大型カードボタン（郵送：レトロカー、速達：鳥、置き手紙：方位磁針）が並びます。
    * 物理的距離から算出された所要時間が、手書き風フォントの吹き出しで**「約3時間後」「約2日後にお届け」**とおおまかな目安時間としてポップアップ表示されます。
    * **グループ置き手紙における「1人だけが受信できる機能」の選択**:
      * グループ宛てに「置き手紙（Dropモード）」を設置する際、以下のオプションを選択可能にします。
        1. **全員受信モード**: グループメンバー全員がそれぞれのタイミングで回収できるモード。
        2. **先着1人限定（早い者勝ち）モード**: グループメンバーのうち、誰か最初の1人がその場所に近づいて手紙を発見・回収した時点で、他のメンバーのレーダーからは消失し、回収できなくなるモード。
        3. **グループ内特定の1人指定モード**: グループという文脈・繋がりを維持したまま、グループ内の特定のメンバー1人だけを指定してその場所に設置するモード（指定された本人しかレーダーに表示されず、回収も不可）。
    * 「この方法でポストに入れる」ボタンをタップすると、画面いっぱいに描かれたレトロなポストの投函口へ封筒が滑り落ちていく送信完了アニメーションが流れます。
    * 送信完了後、**隠されていた下部ナビゲーションバーがスライドインで元の位置に復活**し、自動的に「ポスト」タブへと画面が引き戻されます。

#### ④ フレンド（フレンド・グループ管理・交換日記デザイン・お題）
従来のフレンド一覧と、新設されるグループ管理機能を統合したコミュニケーションハブ画面です。
* **フレンド管理**: 「フレンドID検索」「QRコードスキャン」「近距離通信（BLE等を用いた近接端末マッチング）」によるフレンド申請・承認・一覧表示。
* **グループ作成（招待制）**: 
    * 任意のフレンドを複数人選択して、招待制の「グループ」を作成できます。
    * グループ宛てには「メンバー全員への一斉送信手紙」や、上述した「グループメンバーなら誰でも開けられる置き手紙 / 1人限定置き手紙」の作成・設置が可能になります。
* **交換日記の同時生成機能**:
    * グループ作成時に「**交換日記も作成する**」チェックボタンを提供します。
    * LazyLoadや近代的なトランジションを用いてグループ作成フローと統合します。
        * **交換順番の設定**: メンバーの一覧をドラッグ＆ドロップで並び替え、近代SNS同様の直感的な操作で、日記を回すローテーション順序（例: A → B → C → A...）を決定します。
        * **交換日記のデザイン選択**: 日記の表紙やベースとなるノートのテーマデザインを選択します。
    * 設定完了後、最初の執筆者が「手紙作成」と同様の流れで第1ページ目を執筆し、送信することで交換日記リレーがスタートします。

※将来的なテーマ追加時は、これらのスロットに対応するカラー（例：サクラモード、抹茶モードなど）を別定義の `ThemeExtension` にパッケージングして追加します。

#### ② タイポグラフィとUIパーツの共通ルール
* **フォントスタイル**: アプリ内のデフォルトのナビゲーションや近代SNSライクなシステム文字には、Flutter標準の `Noto Sans JP` を使用。手紙本文や情緒的メッセージには、動的に読み込まれる `Noto Serif JP`（明朝）やオリジナル手書きフォントを割り当てます。
* **角丸（BorderRadius）**: 直線的な角（`0`）は極力排除し、カードやボタンには `BorderRadius.circular(16)` または `24` を適用して、手になじむ丸みを演出します。
* **影（BoxShadow）**: 強い黒い影は使わず、`color: HexColor('#2C2C2C').withOpacity(0.05)`, `blurRadius: 12`, `offset: Offset(0, 4)` のような、光が優しく拡散したような淡い影で紙の「浮き出し感」を表現します。

#### ③ アニメーションイージング・秒数（Motion Spec）
Flutterでのアニメーション実装時は、機械的な直線移動（Linear）を禁止し、すべて生物的な揺らぎを感じる `Curves` を使用します。

* **紙をベラベラめくるエフェクト**: `Duration(milliseconds: 400)` / `Curves.easeInOut`
* **便箋を折りたたむ・封緘モーション**: `Duration(milliseconds: 800)` / `Curves.anticipate`（一度広がってからシュッと閉じる）
* **ポストへの投函アニメーション**: `Duration(milliseconds: 600)` / `Curves.bounceOut`（ポストの底に落ちた際のわずかな跳ね返り）
* **ゴミ箱への破棄エフェクト**: `Duration(milliseconds: 500)` / `Curves.easeInBack`（シュッと縮小しながら回転して消える）

### 2.2 フォントデータのストレージ＆通信量最適化設計 (Cloudflare R2 連携)

フォントデータによる通信量とサーバーコストを徹底的に削減するため、大容量のファイルホスティングに優れた **Cloudflare R2** をストレージとして確定適用します。

* **サブセットフォント/ストローク差分同期の確定採用**:
* ユーザーのオリジナルフォントを読み込む際、数MBに及ぶ日本語の全文字入りTTF/OTFファイルを丸ごと転送することは一切禁止します。
* **実装アプローチ**: データベース（Supabase）上には文字単位の軽量ベクターデータ（`vector_data`: 数百バイトのSVG形式）のみを保管。手紙送信時に、アプリ側またはSupabase Edge Functionsで「その手紙に実際に使用されている文字データのみ」を抽出・パッケージングした極小のサブセットファイルを自動でビルド・生成し、Cloudflare R2 へ直接アップロードして転送します。
* これにより、フォント関連の通信・ファイルサイズは手紙1通あたりわずか数KB〜数十KBへと極小化されます。



### 2.3 セキュリティ・悪用防止・コスト最適化設計

#### ① Supabase 行レベルセキュリティ (Row Level Security: RLS) の徹底

PostgreSQL標準かつ極めて強力なRLSポリシーを適用し、他人の手紙の盗聴や改ざんを完全に遮断します。

* **usersテーブル**:
```sql
ALTER TABLE public.users ENABLE ROW LEVEL SECURITY;
CREATE POLICY "ユーザーは自身のプロフィールのみ書き込み可能" ON public.users FOR UPDATE USING (auth.uid() = id);
CREATE POLICY "認証済みユーザーは全員のプロフィールを閲覧可能" ON public.users FOR SELECT USING (auth.role() = 'authenticated');

```


* **lettersテーブル**:
```sql
ALTER TABLE public.letters ENABLE ROW LEVEL SECURITY;
CREATE POLICY "手紙は送信者または宛先またはグループメンバーのみ閲覧可能" ON public.letters FOR SELECT 
USING (
  auth.uid() = sender_id OR 
  auth.uid() = recipient_id OR 
  (recipient_group_id IS NOT NULL AND auth.uid() IN (SELECT user_id FROM public.group_members WHERE group_id = recipient_group_id))
);

```



#### ② Cloudflare R2 セキュリティと有効期限付き署名（Presigned URLs）

* R2バケットは原則完全非公開（Private）に設定します。
* 添付写真（5MB以下制限、アプリ側で圧縮）や生成されたサブセットフォントへのアクセスが必要な場合、FlutterクライアントはSupabase Edge Functionsを介して、有効期限付きの **R2 署名付きURL（Presigned URL: 有効期限15分など）** を動的に取得します。これにより、直リンクによる画像の不正流出、スクリプトアタック、および無駄なダウンロードによる転送量の乱費を防ぎます。

#### ③ コスト削減のためのクエリ・キャッシュ戦略

* クライアント側（Flutter）でSupabaseのデータをローカルの SQLite（`hive` や `isar` などの永続性ローカルディスクキャッシュ）へ積極的にミラーリングします。
* ポスト内の過去の手紙や交換日記の一覧、フレンド一覧などは、一度取得したデータはローカルキャッシュから即時読み込み、データの更新差分のみをRealtime機能またはタイムスタンプチェックで同期することで、Supabase Databaseへの無駄なAPIリクエストを劇的に削減します。
* **グループ置き手紙の出現クエリの最適化**: 置き手紙の出現クエリは、PostgreSQLのGIS機能（または簡易な緯度経度矩形判定）とGeohashを併用し、自身の位置周辺のレコードのみを限定抽出。先着1人限定モード（`first_come`）の場合は、`cardinality(discovered_user_ids) = 0` の条件を加え、誰か1人が拾った時点で自動的に他のメンバーのレーダー用クエリ対象外となるようインフラレイヤーでフィルタリングします。

### 2.4 バックグラウンド処理・到着判定と通知の自動化

* **Supabase Cron (pg_cron) & Edge Functions による自動リレー判定**:
* 手紙や交換日記が送信された際、到着予定時刻（`estimated_delivery_at`）をトリガーとするバッチ処理を効率化するため、Supabase上の **pg_cron** もしくは定期実行型 **Cloudflare Workers (Cron Triggers)** を導入します。
* 毎分稼働する軽量なCronが「現在時刻 ≧ `estimated_delivery_at` かつ status = 'transit'」のレコードを自動検知。
* 条件に該当するアイテムを発見した場合、バックエンドロジック（Edge Functions）が自動的に status を `delivered`（個人の場合）に更新、または交換日記を `overdue` チェック状態に移行。
* 同時に、バックエンドから通知配信用ゲートウェイを経由して宛先ユーザーの端末へプッシュ通知を即時デリバリーします。これにより、完全バックグラウンドでの1分精度のクエリ・通知システムを稼働させます。
